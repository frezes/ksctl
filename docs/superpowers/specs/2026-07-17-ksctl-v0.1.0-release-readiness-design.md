# ksctl v0.1.0 Release Readiness Design

## Goal

Prepare the repository for its first public GitHub release, `v0.1.0`, by
removing known security and correctness blockers and adding a reproducible,
reviewable release pipeline.

## Release Scope

The release publishes both supported entrypoints:

- `ksctl`, the standalone CLI;
- `kubectl-ks`, the kubectl plugin executable invoked as `kubectl ks`.

Both binaries are built for the following targets:

| Operating system | Architectures |
| --- | --- |
| Linux | `amd64`, `arm64` |
| macOS (`darwin`) | `amd64`, `arm64` |

Each binary and target combination has its own tar.gz archive. GoReleaser
produces one SHA-256 checksum file covering all published archives. The Git tag
is injected into both binaries with a linker flag so `version` reports
`v0.1.0` instead of `dev`.

The release does not include Windows binaries, container images, Homebrew or
Krew manifests, Linux packages, SBOMs, artifact signatures, SLSA provenance,
or macOS code signing and notarization.

## Security and Correctness Changes

### Token cache paths

Fleet and User names are untrusted path components. A cache-only encoder, kept
separate from the human-readable `SafeName` helper, maps each UTF-8 input byte
injectively. ASCII letters, digits, `_`, `-`, and ordinary dots pass through;
every other byte becomes `‾HH`, where `HH` is uppercase hexadecimal. A literal
`‾` therefore becomes `‾7E`. The empty string becomes the otherwise
unproducible bare `‾`. If the complete encoded value would be `.` or `..`, each
dot becomes `‾2E`.

Examples are exact: `prod` remains `prod`, `a/b` becomes `a‾2Fb`, `a:b`
becomes `a‾3Ab`, `.` becomes `‾2E`, and `..` becomes `‾2E‾2E`. The mapping
must never create a nested path, and existing paths such as `prod/admin.json`
remain stable.

Tests cover traversal values, separator characters, escape characters, and
two inputs that previously collapsed to the same sanitized name. Every
computed path must remain under the configured token cache directory.

### Credential persistence

Configuration and token cache writes use a same-directory temporary file,
apply mode `0600`, close the file, and atomically rename it over the target.
Parent directories continue to use mode `0700`. Every save enforces the target
file mode even when an existing file had broader permissions. Temporary files
are removed after any failed write, close, chmod, or rename step.

### Hidden password input

The prompting boundary distinguishes normal line input from password input.
When stdin is a terminal, password input uses the terminal password API and
prints a trailing newline without echoing the entered value. Tests use a fake
prompter rather than a real terminal and verify that the password-specific
method is selected. Non-interactive behavior remains unchanged: all required
values must be supplied explicitly.

OAuth failures continue to return a generic operation error and must not expose
the response body, password, access token, or refresh token.

### Redacted config output

`ksctl config view` redacts `password`, `bearerToken`, and TLS private
`keyData` values by default. The keys remain visible with a stable
`<redacted>` marker so users can distinguish configured credentials from
missing credentials. `bearerTokenFile`, certificates, CA data, endpoints, and
non-sensitive context metadata remain visible.

An explicit `ksctl config view --raw` prints the stored representation without
redaction. Help text warns that raw output can contain credentials. Redaction
operates on a copied configuration and never mutates the object that could
later be saved.

### kubectl plugin help

The plugin root command keeps the executable name `kubectl-ks` and sets Cobra's
`CommandDisplayNameAnnotation` to `kubectl ks`. The visible parent name is used
when constructing inherited kubectl commands, so root usage, subcommand usage,
examples, help, and completion text consistently identify `kubectl ks`.

### Unsupported workspace flag

The current `--workspace` flag only stores a string and never changes a request.
It is removed from the v0.1.0 command surface and from both READMEs rather than
shipping a non-functional scope selector. Workspace selection can return in a
later release after its server-side request semantics and tests are defined.

### Go module consistency

`go mod tidy` promotes directly imported `k8s.io/klog/v2` from the indirect
dependency block to the direct block. CI verifies `go mod tidy -diff` so module
metadata drift blocks merging and release.

## Build and Release Configuration

The Makefile remains the local entrypoint:

- `make build` builds both binaries in `bin/`;
- `make test` runs the normal test suite;
- `make verify` runs formatting, module consistency, vet, normal tests, race
  tests, and both builds;
- `make clean` removes both local binaries.

GoReleaser uses configuration schema version 2. Its two build definitions use
`CGO_ENABLED=0`, `-trimpath`, Linux/macOS, amd64/arm64, and the same version
linker flag. Snapshot builds exercise the complete archive matrix without
publishing.

## GitHub Actions

The CI workflow runs for pull requests and pushes to `main` or `develop`. It
uses the Go version declared by `go.mod` and executes the same release gates as
`make verify`. The workflow has only `contents: read` permission.

The release workflow runs only for tags matching `v*`. It checks out complete
history, sets up the `go.mod` Go version, repeats the critical test and module
checks, and then runs GoReleaser. Only the release job receives
`contents: write`; it authenticates with the repository-scoped
`GITHUB_TOKEN`.

All referenced GitHub Actions are pinned to verified full commit SHAs with a
nearby version comment. GoReleaser itself is pinned to immutable release
`v2.17.0` rather than a moving `latest` or broad version range.

A failed check, invalid GoReleaser configuration, non-buildable target, or
archive error stops the workflow before publication. The workflow does not
move or overwrite an existing release tag.

## Documentation and Changelog

`CHANGELOG.md` records the `v0.1.0` feature set and the release-readiness fixes
using Keep a Changelog-style headings. Both READMEs document:

- supported platforms and architectures;
- archive download and installation;
- SHA-256 verification;
- standalone and kubectl-plugin invocation;
- source builds and local verification;
- the fact that password input is hidden;
- the default redaction and explicit risk of `config view --raw`;
- the absence of a supported workspace selector in v0.1.0.

The repository preparation creates no Git tag, GitHub Release, push, or remote
repository setting change. Publishing `v0.1.0` remains an explicit maintainer
action after review.

## Testing and Acceptance

Implementation follows test-driven changes for each behavior change. Focused
tests prove:

- cache paths cannot escape and distinct names do not collide;
- credential saves are atomic in the successful path and enforce permissions;
- password prompts call hidden input and preserve non-interactive behavior;
- default config output redacts every defined secret while `--raw` preserves
  the original values;
- `kubectl-ks` root and subcommand help display `kubectl ks`;
- the workspace flag is absent;
- linker-injected versions are reported by a built release candidate.

Final acceptance requires all of the following to pass from a clean worktree:

```text
gofmt check
go mod tidy -diff
go mod verify
go vet ./...
go test ./... -count=1
go test -race ./... -count=1
build ksctl and kubectl-ks with release linker flags
GoReleaser configuration check and snapshot build
git diff --check
```

The generated Linux/macOS archive names, file contents, and checksum manifest
are inspected before the repository is declared release-ready.
